### Development monitoring

The following additive alerts use the existing `carewest-intake-ops` action group and
do not change the production severity-1 alerts:

- `carewest-intake-dev-dataverse-failure`: one-minute scheduled query over a five-minute
  window for PII-free `dataverse delivery failed` traces from the dev role name
- `carewest-intake-dev-delivery-failure`: severity-2 Application Insights exception
  metric alert; it fired during the controlled Dataverse outage test
- `carewest-intake-dev-queue-backlog`: severity-2 alert when the isolated queue service
  remains non-empty over a one-hour window, including the Dataverse and poison queues

The queue service metric is aggregate for the isolated data account; using a dedicated
account prevents unrelated production queues from affecting the dev alert.

### Cost and rollback

The dev Flex plan has no always-ready instance and scales to zero. Charges can still
accrue for Function executions, two Standard LRS storage accounts, Log Analytics data
ingestion, and alert-rule evaluation.

To stop intake without deleting evidence, set the dev feature flag to false and mark
the synthetic QR entry inactive. To remove the isolated environment after evidence is
exported, delete the three dev alert rules, dev Function App, dev Flex plan, dev
Application Insights component, dev managed identity, and both dev storage accounts.
Resource deletion is destructive and requires a separate approved cleanup action. The
production Function App, identity, storage accounts, alerts, and feature flag are not
part of that rollback.

## Safety rule

`POST /api/report` and its `safety-intake-delivery` queue are the current production
path. Do not change their settings, payload contract, callback URL, or queue while
rolling out the extensible path.

## Prerequisites

- An approved Power Platform environment, not the personal Developer environment.
- The `Carewest Operations Intake` solution imported with Dataverse tables, security
  roles, connection references, and the notification flow disabled.
- The Azure Function managed identity added to Dataverse as an application user with a
  least-privilege role that can create/update Intake Submission and read Site, Intake
  Program, and QR Entry Point.
- Exact Dataverse logical column names and entity-set names exported from environment
  metadata. Do not infer plural names or publisher-prefixed logical names.

## Required Function settings

Keep the feature flag false until the validation section passes:

```text
CAREWEST_EXTENSIBLE_INTAKE_ENABLED=false
CAREWEST_CONFIG_TABLE_NAME=CarewestIntakeConfig
CAREWEST_DATAVERSE_QUEUE_NAME=carewest-dataverse-delivery
DATAVERSE_ORG_URL=https://<approved-org>.crm.dynamics.com
DATAVERSE_INTAKE_ENTITY_SET=<exact entity-set name>
DATAVERSE_FIELD_MAP_JSON=<secret app setting containing the exact logical-name map>
```

The field-map JSON must define `primaryName`, `submissionId`, `externalReportId`,
`summary`, `description`, `structuredPayloadJson`, `azureDeliveryStatus`, and
`receivedTime`, plus the numeric `azureDeliveryStatusValue`. Optional fields and
lookups are mapped only when their exact logical names and choice values are known.
The current Developer-environment metadata snapshot is recorded in
`infra/dataverse-field-map.development.example.json`; production metadata must be
queried again after the managed solution is imported.

## Deploy and validate while disabled

1. Run `npm test` and `npm run check` from `api/`.
2. Deploy the Function package with `CAREWEST_EXTENSIBLE_INTAKE_ENABLED=false`.
3. Deploy `docs/` to the Static Web App. Its navigation fallback must serve
   `/q/{entryKey}` from `index.html`.
4. Confirm the existing `/api/health` endpoint is healthy.
5. Confirm `GET /api/entry-points/test-cleaning` returns `404` while disabled.
6. Submit a synthetic report through the existing safety form and confirm its outbox
   row reaches `Delivered`. Do not put resident, employee, or medical data in the test.
7. Synchronize only inactive synthetic entry points:

   ```bash
   cd api
   npm run sync:entry-points -- ../infra/entry-points.example.json
   ```

## Controlled development activation

1. Set exact Dataverse organization, entity set, and field map values.
2. Enable the feature flag in a development-only Azure slot or isolated Function App.
3. Activate one synthetic QR Entry Point whose Site, Intake Program, and QR Entry Point
   Dataverse IDs point to synthetic rows.
4. Open `/q/{entryKey}` and verify the site is locked and only allowed report types are
   shown.
5. Submit the same synthetic payload twice with one `submissionId`. Expect one
   Dataverse Intake Submission and the same public report ID.
6. Reuse that `submissionId` with different content. Expect `409` and no second row.
7. Temporarily remove Dataverse access, submit once, restore access, and verify the
   outbox changes from `PendingRetry` to `Delivered` without creating a duplicate.
8. Verify logs contain report IDs and state transitions, but no names or narrative.

## Notification-flow gate

Keep `Submission Assignment Notification` off until it resolves the QR Entry Point,
selects the responsible manager from governed configuration, handles a missing manager,
and uses an IT-owned connection reference. A successful Dataverse insert is not proof
that an email or Teams notification was delivered.

## Production cutover

Import a managed solution into the IT-owned environment, bind corporate connection
references and application identities, repeat every controlled activation test, and
enable one Site plus Intake Program at a time. Keep the safety fallback published until
the business, privacy, records, security, and technical owners sign off. Record every
owner, evidence link and approval date in
[`../governance/PRODUCTION-READINESS-GATE.md`](../governance/PRODUCTION-READINESS-GATE.md).
